About Purr

The Beginning of Booster

Booster is a company that has been making people’s lives easier and more convenient in the past 4 years. The idea is simple: you order a boost via the mobile app and you get high-quality gas at your parking spot while you are away.

In the beginning, we had to make a choice about the DBMS around which we would build Booster: we needed flexibility, high availability and easy development. All of these were offered by MongoDB for free. To make our loyal users happy, we constantly improve our applications which sometimes causes changes in our data structures. MongoDB handles these situations well and lets us focus on different problems.

Booster quickly became part of people’s everyday lives in Silicon Valley, resulting with rapid growth which has not stopped since then.

Need for PostgreSQL

Just like many other companies, we recognized the importance of data analysis as we grew. It helps us increase our productivity and make better decisions for the future. Today, SQL is a must and almost every developer knows how to write basic SQL queries.

Choosing the right RDBMS is also a tough task but we chose Postgres because it offered the most important things we were looking for: it is object-relational, fast and it has support for various data types. Postgres is flexible: the JSON support lets us go schema-less whenever we want to.

Adventures with MoSQL

We used an ETL tool called MoSQL to transfer our data from MongoDB to Postgres. MoSQL is open-source and easy to use: it transfers all of the collections defined in a collection map and syncs with MongoDB for changes. We loved MoSQL but it is no longer maintained and lately some issues appeared more and more frequently: MoSQL crashed when there were schema changes and connectivity issues. This resulted with reimporting millions of documents from the beginning. Before the reimporting has been started, all of the tables were dropped by MoSQL, making us unable to generate reports and plan the next day(s) or week(s). A full reimport took more than 5 hours. As time passed, we were more and more sure that we need to take things into our own hands.

A Real Solution

We created Purr to do what MoSQL did right and handle the issues MoSQL could not handle anymore. Since we relied on MoSQL and did not want to make drastic changes in our app logic, we made Purr except very similar input we already had and provide a MoSQL-like output. We also needed to keep performance in mind and connectivity issues MoSQL could not always handle. Purr has many features that MoSQL doesn't: you can set it to transfer to a custom schema, in case of reimport you can tell it to keep your schema, drop your tables or specific columns, etc. In case of connectivity issues from both MongoDB and Postgres, Purr waits for a while, attempts to reconnect and starts tailing the oplog again. Extra properties: You can include and exclude extra properties that are not defined in your collection map. Leaving only the necessary information decreases transfer time.
